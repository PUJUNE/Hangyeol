# conventions.md — 산출물 양식 규칙

이 문서는 `code-narrator-ko` 스킬이 생성하는 모든 산출물의 양식 표준임. 모든 항목은 사용자 개입 없이 일관 적용됨.

---

## 1. 어미와 문체

- **모든 산출물은 `~음` 체로 통일**. 예: `~확인됨`, `~있음`, `~필요함`, `~판단됨`, `~함`.
  - `~합니다`, `~이다`, `~한다` 류 회피.
  - 단, 사용자 인용·기존 코드 주석·외부 표준 용어는 원형 보존.
- **AI 상투어 금지** (CLAUDE.md): `중요한, 핵심적인, 본질적으로, 결국, 단순히, 명확히, 주목할 만한, 흥미롭게도, 심층적으로` 등을 구체 수치·관찰로 대체.
- **능동태 우선**: "감지됨/제기됨/이루어짐" 류 피동 회피. 행위자가 분명할 때 능동.
- **중·장문 기본**: 40~80자로 절 2~3개, 단문 남용 회피. 같은 길이 문장 3개 이상 연속 금지.

## 2. 라인 주석본 (`03_라인주석/<원본경로>`)

### 기본 규칙

- **모든 의미 있는 줄에 한국어 주석 1줄**. 빈 줄·머리글 주석 등은 그대로 통과.
- **비유 사용 금지**. 비유는 파일별 요약에서만. 라인 주석은 "이 줄이 무엇을 하는지"만 사실적으로.
- **줄당 한 줄로 끝**. 여러 줄 설명이 필요하면 그 의미 단위를 파일별 요약으로 옮김.
- **위치**: 줄 끝 인라인이 기본. 너무 길어 가독성을 해치면 그 줄 위에 한 줄 추가.
- **묶음 표지 주석**: 원본의 구획 주석(`/* ============ HEADER ============ */` 등) 바로 아래에 한국어 묶음 표지 한 줄 추가 권장.

### 파일 타입별 주석 기호

| 확장자 | 주석 기호 | 예시 |
|---|---|---|
| `.html`, `.htm`, `.xml`, `.svg` | `<!-- 주석 -->` | `<meta charset="UTF-8">  <!-- 글자 인코딩 UTF-8 -->` |
| `.css` | `/* 주석 */` | `color: #fff;  /* 글자색 흰색 */` |
| `.js`, `.jsx`, `.ts`, `.tsx`, `.json` | `// 주석` | `let count = 0;  // 카운터 0으로 시작` |
| `.md` | `<!-- 주석 -->` (위 줄) | 표 행 등 인라인 불가능한 경우 위 줄에 |
| `.txt`, `.cfg`, `robots.txt` | `# 주석` | `Allow: /  # 사이트 전체 허용` |
| `.py`, `.r`, `.sh` | `# 주석` | (R&D 확장 시) |
| `.m` (MATLAB), `.jl` (Julia) | `% 주석` (MATLAB), `# 주석` (Julia) | (R&D 확장 시) |

**JSON 주의**: 표준 JSON은 주석을 허용하지 않음. 주석본은 사람이 읽는 학습용 파일이며 파서가 읽을 일이 없으므로 `// 주석` 형태로 추가해도 무방. 원본은 손대지 않음.

### 주석 내용 규칙

- **그 줄이 컴퓨터에게 무엇을 시키는지**를 한국어로 설명.
- 변수·함수·열쇠 이름이 중요할 때 그대로 인용 (백틱·따옴표 없이).
- 화면에 보이는 결과까지 짧게 언급하면 좋음 (예: "흰 글자로 표시").
- 같은 패턴이 반복되면 두 번째부터는 "(동일)" 또는 생략 가능.

## 3. 파일별 요약 (`02_파일별_요약/<원본경로>.md`)

### 표준 4개 절

```markdown
# <파일명> — <한 줄 정체>

## 전체 요약

(한 단락. 비유로 시작 — 호텔/책/공장/자료실 등 analogy-pool에서. 이 파일이 사이트 안에서 어떤 역할을 맡는가, 다른 파일과 어떻게 다른가, 특이점은 무엇인가. 200~400자.)

## 의미 단위 묶음

### ① <묶음 이름> (N~N행)

```<언어>
... 묶음의 핵심 코드 ...
```

- **무엇** — 사실 기반 한 줄 설명.
- **비유** — 그 묶음을 일상 비유로 한 단락. analogy-pool에서.
- **세부 포인트** (선택) — 트리거가 되는 디테일·주의 사항.

### ② <묶음 이름> (N~N행)
...

## 비개발자가 자주 묻는 수정 질문

| 하고 싶은 것 | 어디를 고치는가 |
|---|---|
| ... | ... 행 또는 변수명 |

## 함께 보면 좋은 파일

- `path/to/file` — 한 줄 관계 설명.
```

### 의미 단위 묶음 분할 기준

- 2~5줄 단위가 기본. 너무 잘게 쪼개면 큰 그림이 안 보이고, 너무 크면 비유가 산만해짐.
- 원본의 구획 주석(`/* ============ */`)을 자연스러운 묶음 경계로 사용.
- 묶음 제목은 짧게(예: "head", "히어로", "JS: 슬라이드쇼"). 원어 + 한국어 부제 권장.

### 비유 사용 규칙

- 한 파일 요약 안에서 **하나의 메타포 카테고리만** 일관 사용. 호텔과 공장을 섞지 않음.
- analogy-pool.md의 매핑 표에서 그 파일 타입에 맞는 카테고리 선택.
- 비유는 비유로 끝내지 말고 **구체 수정 지점**(예: "메뉴 항목 추가는 X 배열")으로 마무리하면 학습 효과가 큼.

### 수정 질문 표

- 비개발자가 실제로 묻는 빈도 높은 변경 5~10개를 표로.
- "어디를 고치는가" 칸에 **파일경로 + 행 번호 또는 변수명**까지 명시. 추측 단어 금지.

### 함께 보면 좋은 파일

- 의존 관계가 있는 파일(이 파일이 호출하거나 호출받는 파일)을 3~6개 나열.
- 각 항목 한 줄 관계 설명.

## 4. 상위 두 문서

### 4-1. `01_읽는법.md` (비개발자용 사전)

- 기본 골격은 `readme-seed.md`. 실제 발견된 파일 타입에만 맞춰 절을 추가·축소.
- 표준 절 — (1) 큰 그림 (2) 파일 종류별 역할 (3) 각 언어 빠른 사전 (HTML/CSS/JS/JSON/XML/MD/TXT 중 발견된 것만) (4) 자주 나오는 기호 한 줄 사전 (5) 산출물 읽는 법 (6) 처음 읽는 사람을 위한 권장 순서.

### 4-2. `00_아키텍처.md` (통합 지도, 마지막에 작성)

- 표준 12개 절 — 한 줄 요약 / 가장 큰 그림(N개 층) / 페이지 지도 / 한 페이지 N단계 / 다국어 또는 데이터 처리 방식 / 데이터→페이지 흐름 / 헤더·푸터 또는 공통 골격 / 폴더 분리 구조 / 수정 진입점 4개 그룹 / 자주 일어나는 변경 표 / 정적/동적 한계와 해결 / 비개발자 권장 학습 순서.
- 텍스트 다이어그램 적극 사용 (ASCII 박스·화살표).
- 마지막 부록: 처리한 N개 파일 전체 목록 + 산출물 위치 표.

## 5. 폴더 구조 미러링

`02_파일별_요약/`와 `03_라인주석/`은 **원본 폴더 구조를 그대로 복제**. 예:

```
원본:                              산출물:
foo/                              주석본/
├── index.html                    ├── 02_파일별_요약/
├── assets/                       │   ├── index.html.md
│   └── style.css                 │   └── assets/style.css.md
└── data/users.json               │   └── data/users.json.md
                                  └── 03_라인주석/
                                      ├── index.html
                                      └── assets/style.css
                                      └── data/users.json
```

- 라인 주석본은 **원본과 같은 확장자·같은 파일명** (안의 코드도 거의 같지만 주석이 추가됨).
- 파일별 요약은 **`<원본파일명>.md`**.
- 폴더 단위 통합 요약은 **`_00_<폴더명>_묶음_요약.md`** (preview·slides 같은 동질적 묶음에만).

## 6. TaskList 사용

- 작업 시작 시 다음 작업 단위를 TaskCreate로 만듦:
  - 파일 분류 + 보고
  - 출력 폴더 생성
  - `01_읽는법.md` 작성
  - 메타·데이터 파일 처리
  - 스타일·JS 처리
  - 페이지 처리 (그룹별)
  - 디자인 시스템·참조 자료 처리
  - `00_아키텍처.md` 작성
- 각 작업 시작 시 `in_progress`, 완료 시 `completed`로 갱신. 사용자가 진행도를 시각적으로 확인할 수 있게.

---

## 7. 컨텍스트 관리

- 한 세션에서 끝나지 않을 정도로 큰 폴더(40개 이상 파일·총 10000줄 이상)면 작업 시작 시 사용자에게 안내. 분할 처리 권장.
- 매 5~10 파일마다 진행 상황을 짧게 보고 (TaskList로 충분).
- 한 파일에서 막히면 그 파일은 "요약만"으로 강등 후 진행, 막힌 이유를 보고에 기록.
