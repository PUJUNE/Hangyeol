# output-templates.md — 산출물 템플릿

이 문서는 라인 주석본·파일별 요약·`01_읽는법.md`·`00_아키텍처.md`의 **표준 템플릿**임. 모든 산출물은 이 양식을 따름.

---

## 1. 라인 주석본 템플릿

### HTML

```html
<!DOCTYPE html>                                                          <!-- HTML5 문서 선언 -->
<html lang="ko">                                                          <!-- 기본 언어 한국어 -->
<head>                                                                   <!-- 페이지 정보 영역 시작 -->
<meta charset="UTF-8" />                                                 <!-- 글자 인코딩 UTF-8 (한글 안전) -->
...
```

### CSS

```css
/* ============ HEADER ============ */                                  /* 사람용 묶음 표시: 헤더 영역 */
.bp-header {                                                             /* 헤더 본체 */
  position: fixed; inset: 0 0 auto 0; z-index: 100;                      /* 위 고정, 위·좌·우 0, 다른 요소 위 */
  display: flex; align-items: center; justify-content: space-between;    /* 가로 배치, 양 끝 정렬 */
  ...
}
```

### JavaScript

```js
(function () {                                                          // 즉시 실행 함수 시작 — 변수 격리
  const BASE = document.body.dataset.base || '.';                       // <body data-base="..."> 값을 BASE에 저장
  ...
```

### JSON (라인 주석본은 사람용이라 표준 JSON 파서가 못 읽어도 무방)

```js
{                                       // JSON 객체 시작
  "ko": {                               // 한국어 사전 시작
    "nav.products": "제품",              // 열쇠 nav.products → 한국어 번역
    ...
```

### 정렬 규칙

- 주석은 충분한 공백 후 (8~16칸) 시작해 시각적 정렬.
- 한 화면 가로 폭 안에 코드+주석이 들어가도록 주석 길이 30~50자 권장.

## 2. 파일별 요약 템플릿 (표준)

```markdown
# <파일명> — <한 줄 정체>

## 전체 요약

이 파일은 **<한 줄 정체 자세히>**임. 비유하면 **<analogy-pool에서 선택한 메타포>**임. <어떤 역할을 사이트 안에서 맡는가, 다른 파일과 어떻게 다른가, 특이점은 무엇인가를 한 단락 200~400자.>

<선택: 두 번째 단락으로 기술적 구조나 한계 설명.>

## 의미 단위 묶음

### ① <묶음 이름> (N~N행)

```<언어>
... 묶음의 핵심 코드 (5~15줄 정도) ...
```

- **무엇** — 사실 기반 한 줄 설명.
- **비유** — 일상 비유 한 문장. analogy-pool에서 선택한 카테고리 일관 적용.
- **<선택: 세부 포인트 이름>** — 트리거가 되는 디테일·주의 사항·왜 이렇게 했는가.

### ② <묶음 이름> (N~N행)

(같은 양식)

## <선택: 추가 절>

- **함수 구조 설명**, **데이터 흐름**, **다른 페이지와의 연결**, **알려진 한계** 등 파일 특성에 맞춰.

## 비개발자가 자주 묻는 수정 질문

| 하고 싶은 것 | 어디를 고치는가 |
|---|---|
| <자주 있는 변경 5~10개> | <파일경로 N행 또는 변수명> |

## 함께 보면 좋은 파일

- `path/to/file` — 한 줄 관계 설명.
- `path/to/file` — 한 줄 관계 설명.
```

## 3. 요약만 파일의 추가 양식

```markdown
# <파일명> — <한 줄 정체>

> **이 파일은 라인 주석을 만들지 않습니다.** <라인 주석 생략 이유 — production 경로 또는 자동 생성 사유>

## 전체 요약

(표준 양식과 동일.)

## <선택: production 버전과의 차이 표>

| 항목 | 이 파일 | production 버전 |
|---|---|---|
| ... | ... | ... |

## <선택: 같은 폴더의 다른 참조 파일>

(소스/백업 파일들이 모여 있는 경우)

## 의미 단위 묶음 (대분류만)

(전체 라인을 다루지 않고 큰 구조만. 라인 단위 디테일은 production 버전 참조 안내.)

## <표준 마무리 절들>

(수정 질문 표 + 함께 보면 좋은 파일)
```

## 4. `01_읽는법.md` 템플릿

```markdown
# 01. 코드 읽는 법 — 비개발자를 위한 사전

이 문서는 `주석본/` 안의 모든 파일을 읽기 전에 한 번 읽어두는 안내서임.

---

## 1. 큰 그림: <대상 코드의 종류>가 화면에 뜨는 과정 (또는 동작하는 과정)

(서버-클라이언트, 정적-동적, 또는 R&D 코드의 경우 입력-처리-출력 흐름.)

---

## 2. 파일 종류별 역할

| 확장자 | 비유 | 하는 일 |
|---|---|---|
| `.html` | 건물의 뼈대 | 화면에 무엇을 보여줄지 |
| `.css` | 페인트와 가구 배치 | 어떻게 꾸밀지 |
| `.js` | 전기 배선과 자동문 | 사용자가 클릭·스크롤하면 무엇이 움직이는지 |
| `.json` | 창고의 재고 장부 | 표·목록 형태의 데이터 |
| `.xml` | 옛날 양식 장부 | 검색엔진이 읽기 좋은 데이터 |
| `.md` | 워드 문서의 초간단판 | 설명서 |
| `.txt` | 종이 메모 | 사람·기계 모두 읽는 평문 |

(실제 발견된 확장자만 포함. 발견 안 된 건 빼서 학습 부담 줄임.)

---

## 3. <언어 1> 빨리 읽는 법

(예: HTML)

(태그 사전 표 + 자주 나오는 속성 + 핵심 사고방식.)

## 4. <언어 2> 빨리 읽는 법

(예: CSS)

(선택자 사전 + 단위 안내.)

## 5. <언어 3> 빨리 읽는 법

(예: JavaScript)

(변수·함수·이벤트 사전.)

(... 발견된 모든 주요 언어에 대해 절 추가 ...)

---

## N. 자주 나오는 기호 한 줄 사전

| 기호 | 어디서 | 뜻 |
|---|---|---|
| ... | ... | ... |

---

## N+1. 이 폴더의 산출물 읽는 순서

1. 이 문서 (`01_읽는법.md`) — 한 번 읽고 옆에 둠
2. `00_아키텍처.md` — 사이트 전체 지도
3. `02_파일별_요약/<원본경로>.md` — 한 파일을 통째로 이해할 때
4. `03_라인주석/<원본경로>` — 한 줄 한 줄이 무엇을 하는지 봐야 할 때 (디버깅·수정 시점)

### 파일별 요약 읽는 법

(전체 요약 → 의미 단위 묶음 구조 안내.)

### 라인 주석본 읽는 법

(주석 위치·내용·예시.)

---

## N+2. 처음 읽는 사람을 위한 권장 순서

1. 이 문서 통독
2. `00_아키텍처.md` 통독
3. 가장 큰 페이지(<발견된 가장 큰 HTML 파일>)의 요약 통독
4. 궁금한 줄이 있으면 라인 주석본 펴서 확인
5. 같은 방식으로 다른 페이지로 확장

이 순서대로 읽으면 **한 줄도 빼놓지 않고 코드가 굴러가는 원리**가 머리에 잡힘.
```

## 5. `00_아키텍처.md` 템플릿 (12개 절)

```markdown
# 00. 사이트(또는 시스템) 전체 아키텍처

이 문서는 <대상 폴더 이름>의 N개 파일이 어떻게 맞물려 한 사이트(또는 시스템)가 굴러가는가의 통합 지도임.

---

## 1. 한 줄 요약

> <한 줄로 시스템 전체를 압축. 4개 층, 18개 페이지 같이 숫자로.>

---

## 2. 가장 큰 그림: N개 층

(텍스트 다이어그램으로 4개 층 또는 5개 층의 의존 관계.)

각 층의 역할 한 줄.
이 분리의 가치 (수정 진입점 단일화).

---

## 3. <페이지/모듈/단계> 지도

(N개 페이지의 폴더 트리 + 각 페이지의 중요도·URL.)

---

## 4. 한 페이지(또는 한 동작)가 화면에 뜨는 N단계

(시간 순으로 사용자 클릭 → 서버 → 브라우저 처리 → 완성된 화면.)

---

## 5. <시스템 특유의 처리 방식> (예: 다국어, 데이터 흐름, 동적 라우팅)

(이 시스템에서 두드러진 패턴 1~3가지.)

---

## 6. 데이터 → 페이지의 흐름 (구체 예시)

(가장 중요한 데이터 흐름 한 가지를 따라가며 N개 페이지에 동시에 어떻게 등장하는지.)

---

## 7. 공통 골격이 모든 페이지에 자동으로 들어가는 마법

(헤더·푸터 자동 주입 또는 공통 모듈 임포트 패턴.)

---

## 8. <폴더 분리 또는 모듈 구분> 구조

(운영 코드와 디자인 시스템의 분리, 또는 R&D 코드의 분석/유틸 분리.)

---

## 9. 모든 파일을 N개 그룹으로 다시 분류 (수정 진입점 관점)

(콘텐츠 변경 / 디자인 변경 / 동작 변경 / 구조 변경의 4 그룹.)

---

## 10. 자주 일어나는 변경의 진입점 한눈 보기

| 변경 요청 | 어디를 고치는가 | 영향 범위 |
|---|---|---|
| ... | ... | ... |

---

## 11. 이 시스템이 <정적/동적/모놀리식 등>인 이유와 한계

(현재 아키텍처의 정체성과 한계점, 단계적 해결 방향.)

---

## 12. 비개발자가 시스템과 처음 대화할 때의 권장 순서

1. 이 문서와 `01_읽는법.md` 통독
2. 메인 페이지 따라가기
3. "내가 바꾸고 싶은 한 가지" 정하고 10번 표에서 진입점 찾기
4. 그 파일의 요약 → 라인 주석에서 정확한 줄 번호
5. AI에게 "N행을 이렇게 바꿔줘"라고 정확한 의지 표현

---

## 부록: 파일 N개 전체 목록 (산출물 위치 표시)

| # | 원본 파일 | 라인 주석 | 파일 요약 |
|---|---|---|---|
| ... | ... | ✓ 또는 — | ✓ 또는 — |
```
